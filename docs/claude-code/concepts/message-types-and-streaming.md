# Message Types and Streaming

## Overview
Claude Code uses a rich message type hierarchy to represent every event in a conversation -- from user prompts and assistant responses to tool results, system notifications, progress updates, and streaming events. Understanding these types is essential for working with the query loop, building UI components, and integrating via the SDK.

## Mechanism

### Core Message Types
The `Message` union type (from `types/message.ts`, re-exported throughout the codebase) encompasses all conversation-level events. The primary variants are:

#### `UserMessage`
Represents user input or tool results sent back to the API.
- `type: 'user'`
- `message.content`: an array of `ContentBlockParam` (text, images, or `tool_result` blocks)
- `toolUseResult`: optional string summarizing a tool result
- `sourceToolAssistantUUID`: links tool results back to the assistant message that requested them
- Created via `createUserMessage()` in `utils/messages.ts`

#### `AssistantMessage`
Represents a model response from the Anthropic API.
- `type: 'assistant'`
- `message.content`: an array of `ContentBlock` (text, `tool_use`, `thinking`, `redacted_thinking` blocks)
- `apiError`: optional error code (e.g., `'max_output_tokens'`) for recovery handling
- `uuid`: unique identifier for linking tool results
- Carries usage metadata for token tracking

#### `SystemMessage`
A family of system-generated informational messages that are not sent to the API but displayed in the UI or used for internal bookkeeping. Subtypes include:
- `SystemInformationalMessage` -- general info/warnings shown to the user
- `SystemAPIErrorMessage` -- API error notifications
- `SystemCompactBoundaryMessage` -- marks where context compaction occurred
- `SystemMicrocompactBoundaryMessage` -- marks microcompact boundaries
- `SystemLocalCommandMessage` -- output from slash commands
- `SystemMemorySavedMessage` -- confirmation of memory writes
- `SystemStopHookSummaryMessage` -- summary of stop hook results
- `SystemTurnDurationMessage` -- timing information for a turn
- `SystemBridgeStatusMessage` -- IDE bridge connection status
- `SystemAwaySummaryMessage` -- summary shown when returning from away
- `SystemPermissionRetryMessage` -- retry guidance after permission issues
- `SystemScheduledTaskFireMessage` -- scheduled task execution notifications
- `SystemAgentsKilledMessage` -- notification when background agents are terminated
- `SystemApiMetricsMessage` -- API call metrics

All system messages share `type: 'system'` and a `level` field (`'info'`, `'warning'`, `'error'`). Created via `createSystemMessage()`.

#### `ProgressMessage`
Represents intermediate progress from a running tool.
- `type: 'progress'`
- Contains tool name, progress percentage, and status text
- Displayed as spinner/progress indicators in the UI
- Transient: not persisted in conversation history

#### `AttachmentMessage`
Injects context (memory files, skill content, hook results) into the conversation without user action.
- `type: 'attachment'`
- Contains attachment content (memory, hook output, permission decision context)
- Created via `createAttachmentMessage()` and gathered by `getAttachmentMessages()`
- Deduplicated by `filterDuplicateMemoryAttachments()`

#### `ToolUseSummaryMessage`
A post-turn summary of tool calls made during the assistant's response.
- Generated by `generateToolUseSummary()` after tool execution completes
- Created via `createToolUseSummaryMessage()`
- Used for compact context representation in long conversations

#### `TombstoneMessage`
A placeholder that replaces removed or compacted messages.
- Preserves the structural position of a message after compaction
- Ensures tool_use / tool_result pairing remains valid

### Streaming Events

#### `StreamEvent`
Represents incremental tokens arriving from the API during streaming.
- `type: 'stream'`
- Carries partial content blocks as they arrive
- Consumed by the REPL UI for real-time rendering
- Yielded by the `query()` generator alongside full `Message` objects

#### `RequestStartEvent`
Signals the beginning of a new API request within the query loop.
- `type: 'stream_request_start'`
- Yielded before each API call so the UI can show a loading state
- Useful for distinguishing between multiple API calls within a single turn (e.g., after tool execution)

### Message Flow Through the System

```
User Input
    |
    v
processUserInput() --> UserMessage
    |
    v
query() loop
    |-- yields RequestStartEvent
    |-- calls Anthropic API (streaming)
    |-- yields StreamEvent (incremental tokens)
    |-- yields AssistantMessage (complete response)
    |
    |-- if tool_use blocks present:
    |   |-- runTools() executes tools
    |   |   |-- yields ProgressMessage (per-tool progress)
    |   |   |-- permission check via canUseTool
    |   |-- yields UserMessage (tool_result)
    |   |-- yields AttachmentMessage (memory, skills)
    |   |-- yields ToolUseSummaryMessage
    |   |-- loop continues
    |
    |-- if no tool_use (end_turn):
    |   |-- yields SystemMessage (turn duration, etc.)
    |   |-- returns Terminal
    |
    v
REPL UI / SDK consumer
```

### Message Normalization
Before sending messages to the API, `normalizeMessagesForAPI()` transforms the internal message array:
- Strips system messages, progress messages, and tombstones
- Validates image attachments (`validateImagesForAPI`)
- Ensures strict tool_use / tool_result pairing
- Handles thinking block preservation rules (thinking blocks must not be the last in a sequence and must be preserved within an assistant trajectory)

### Normalized Message Types
For internal processing, the codebase defines normalized variants:
- `NormalizedMessage` -- `Message` after filtering out progress and system messages
- `NormalizedAssistantMessage` / `NormalizedUserMessage` -- type-narrowed variants
- `MessageWithoutProgress` -- excludes `ProgressMessage` from the union
